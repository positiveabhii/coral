# Coral Source DSL v4 PRD (Surface-First Projection IR)

## Summary

Coral source specs currently use a table-oriented authoring model. A source
manifest chooses one top-level `backend` and then hard-codes the SQL surfaces
that Coral exposes through `tables:` and `functions:`. For HTTP sources, those
same table/function entries also declare provider request details, response
extraction, filters, arguments, columns, and pagination.

That model works for curated, hand-authored sources, but it does not scale to
providers that already publish authoritative API descriptions such as OpenAPI or
GraphQL schemas. It also forces a source to be represented by one implementation
surface even when a real provider exposes several useful surfaces, such as
GitHub REST and GitHub GraphQL.

DSL v4 introduces a surface-first source model:

1. The manifest declares one or more upstream surfaces and how to call them.
2. Coral imports each surface description into a stable semantic IR.
3. Coral generates SQL projections from that IR by default.
4. Manifest authors may apply projection overrides to curate names, visibility,
   column shape, descriptions, input exposure, and search/detail hints.
5. Runtime execution remains surface-specific. The IR is a projection substrate,
   not a generic physical executor abstraction.

The core design split is:

- **Surface type** answers how Coral talks to the provider: OpenAPI/REST,
  GraphQL, MCP, file scan, or a future protocol.
- **Semantic IR** answers what capabilities the provider exposes in a normalized
  vocabulary that projection generation can use.
- **Projection catalog** answers what SQL tables/functions Coral publishes.
- **Physical planners** answer how a projection is executed for one concrete
  surface type.

The first implementation wave is OpenAPI-first. A v4 manifest can declare
OpenAPI surfaces, Coral imports pinned OpenAPI descriptions during source add,
materializes a semantic IR plus generated projection catalog to local app state,
and later loads the installed source without network access or re-importing the
OpenAPI document.

Current `dsl_version: 3` manifests remain supported unchanged.

## Goals

- Add DSL v4 manifests that declare API surfaces instead of hard-coded
  `tables:` and `functions:` arrays.
- Remove top-level `backend` from v4 manifests. Runtime implementation type is
  declared per surface.
- Import a stable semantic IR from provider-supplied OpenAPI descriptions.
- Use the semantic IR as the common substrate for automatic SQL projection
  generation.
- Allow projection overrides for curation without requiring authors to redeclare
  provider operations, request bindings, response paths, or protocol execution
  details.
- Materialize imported IR and generated projections to local app state during
  source add so normal query-time loads do not fetch or parse upstream
  descriptions.
- Preserve v3 parser, schema, install, and runtime behavior during migration.
- Execute a narrow GitHub issues slice end to end through a v4 OpenAPI surface.
- Keep the implementation aligned with current crate boundaries: source-spec
  parsing in `coral-spec`, app-state materialization in `coral-app`, and
  physical execution in `coral-engine`.

## Non-Goals for this wave

- Convert existing v3 manifests automatically.
- Replace the existing v3 HTTP, JSONL, or Parquet backends.
- Add a full GraphQL importer or GraphQL executor. The IR must leave room for a
  GraphQL fast-follow, but this wave ships OpenAPI execution only.
- Expose mutating operations as SQL affordances. Coral remains read-only for
  generated projections in this wave.
- Infer canonical cross-surface entity identity. Imported entity candidates are
  useful evidence for projection generation, not proof that two surfaces model
  the same logical provider object.
- Build a complete source-authoring UI or broad projection editor.
- Solve every naming, table-vs-function, search ranking, or relationship
  heuristic. The first implementation must be deterministic, documented, and
  overrideable.

## Explicit Anti-Goals

- **Do not hand-author provider operations in v4 manifests.** A v4 manifest must
  not declare provider paths, methods, parameter mappings, response paths, or
  pagination details for imported OpenAPI operations. Those facts come from the
  upstream API description and importer.
- **Do not make the semantic IR the physical executor abstraction.** The engine
  must still know whether a projection resolves to REST/OpenAPI, GraphQL, MCP,
  or another surface type when it builds a physical plan.
- **Do not require normal source authors to inspect generated IR before a source
  can be installed.** Projection generation must have useful defaults.
- **Do not add a synthetic top-level backend for v4.** In v4, `dsl_version`
  selects the v4 manifest shape and each surface declares its own `type`. There
  is no top-level `backend` field in the v4 schema.

The only opinionated authoring surfaces in a v4 manifest are:

- the existing source identity/install envelope (`name`, `version`,
  `dsl_version`, `description`, inputs, and test queries),
- `surfaces:` for upstream descriptions and surface runtime configuration,
- optional `projection_policy:` for broad generation behavior,
- optional `projection_overrides:` for curation of generated SQL affordances.

## Existing System Context

Relevant crates:

- `crates/coral-spec`: parses and validates source manifests. Current parser
  entry points are `parse_source_manifest_yaml` and
  `parse_source_manifest_value`. The typed parse result is
  `ValidatedSourceManifest`.
- `crates/coral-spec/src/schema/source_manifest.schema.json`: current manifest
  schema. It requires `dsl_version: 3`, `name`, `version`, and `backend`.
- `crates/coral-spec/src/common.rs`: defines backend-agnostic manifest pieces,
  including `SourceManifestCommon`, `SourceBackend`, table metadata, filters,
  columns, pagination, and table-function specs.
- `crates/coral-spec/src/backends/http.rs`: parses and validates current HTTP
  manifests. `HttpSourceManifest` contains source-level `base_url`, `auth`,
  `request_headers`, `rate_limit`, `tables`, `functions`, and declared inputs.
- `crates/coral-engine`: compiles validated specs into DataFusion providers.
  Backend dispatch currently lives in `crates/coral-engine/src/backends/mod.rs`.
- `crates/coral-engine/src/backends/http/`: current HTTP runtime. It registers
  manifest-defined tables and functions, builds HTTP requests, applies filter
  pushdown, fetches JSON rows, and maps them into Arrow batches.
- `crates/coral-app`: owns local server composition, source lifecycle, app-state
  layout, installed source catalog, and query-time source loading.
- `crates/coral-app/src/sources/manager.rs`: current source add/import/delete
  workflow. It validates a manifest, persists imported manifests, stores
  variables/secrets, and updates the installed source catalog.
- `crates/coral-app/src/sources/catalog.rs`: resolves bundled and imported
  manifests into `ValidatedSourceManifest`.
- `crates/coral-app/src/state/layout.rs`: current app-state layout. Installed
  imported manifests live under
  `workspaces/<workspace>/sources/<source>/manifest.yaml`; secrets live beside
  them.

Current v3 manifest path:

```text
manifest.yaml
  -> coral-spec ValidatedSourceManifest
  -> coral-engine compile_validated_manifest
  -> backend-specific CompiledBackendSource
  -> DataFusion TableProvider / TableFunctionImpl registration
```

Current HTTP manifests mix projection and execution concerns:

```yaml
name: github
version: 1.1.6
dsl_version: 3
backend: http
base_url: "{{input.GITHUB_API_BASE}}"
auth: ...
request_headers: ...
rate_limit: ...

tables:
  - name: issues
    filters: ...
    request: ...
    response: ...
    pagination: ...
    columns: ...

functions:
  - name: search_issues
    args: ...
    request: ...
    response: ...
    pagination: ...
    columns: ...
```

In v3, `tables:` and `functions:` are both the authored SQL projection catalog
and the authored backend execution plan. DSL v4 separates those concerns.

## DSL v4 Manifest Boundary

DSL v4 keeps the current source identity/install envelope, but replaces
top-level `backend`, `base_url`, `auth`, `request_headers`, `rate_limit`,
`tables`, and `functions` with surface declarations and projection curation.

First-wave manifest shape:

```yaml
name: github
version: 2.0.0
dsl_version: 4
description: >-
  Query GitHub issues through imported provider API descriptions.

inputs:
  GITHUB_API_BASE:
    kind: variable
    default: https://api.github.com
  GITHUB_TOKEN:
    kind: secret

test_queries:
  - SELECT * FROM github.issues LIMIT 1

surfaces:
  - id: rest
    type: openapi
    url: https://raw.githubusercontent.com/github/rest-api-description/<commit>/descriptions/api.github.com/api.github.com.yaml
    sha256: 7f2a...
    base_url: "{{input.GITHUB_API_BASE}}"
    auth:
      type: HeaderAuth
      headers:
        - name: Authorization
          from: template
          template: Bearer {{input.GITHUB_TOKEN}}
    request_headers:
      - name: Accept
        from: literal
        value: application/vnd.github+json
      - name: X-GitHub-Api-Version
        from: literal
        value: "2022-11-28"
    rate_limit:
      extra_statuses: [403]
      remaining_header: x-ratelimit-remaining
      reset_header: x-ratelimit-reset

projection_policy:
  default: derive_read_operations

projection_overrides:
  - match:
      surface: rest
      operation: issues/list-for-repo
    name: issues
    description: Repository issues.
    inputs:
      owner:
        expose_as: filter
        required: true
      repo:
        expose_as: filter
        required: true
    columns:
      include:
        - number
        - title
        - state
        - user.login
        - html_url
      rename:
        user.login: author_login
```

Author-facing changes versus v3:

- Remove top-level `backend` from v4.
- Remove top-level HTTP runtime configuration. Runtime API configuration lives
  on the surface that uses it.
- Remove authored `tables:` and `functions:` from v4.
- Add `surfaces:` as the authoritative list of imported provider surfaces.
- Generate SQL projections from imported IR by default.
- Add optional projection policy and overrides for curation.
- Keep `inputs` at the source level because credentials and install-time
  variables are still source-scoped user configuration.

DSL v4 is additive. Existing `dsl_version: 3` manifests keep their current
schema, including top-level `backend`, `tables`, and `functions`.

## Surfaces and Importers

A surface entry tells Coral where to find an upstream API description, which
importer to apply, and how to call that upstream surface at runtime.

First-wave supported surface type:

- `type: openapi`

Reserved future surface types:

- `type: graphql`
- `type: mcp`

The OpenAPI importer:

1. Resolves the description from a pinned descriptor (`url` plus `sha256`, or a
   local development file with equivalent hash validation).
2. Parses the OpenAPI document.
3. Produces a semantic IR with normalized operations, inputs, output shapes,
   entity candidates, relationships, cardinality, and surface-specific
   execution attachments.
4. Generates a projection catalog from the semantic IR.
5. Applies manifest projection overrides.
6. Writes materialized artifacts to app state during source add.

The semantic IR is intentionally not a public authoring format. It is an
internal, stable-enough artifact for projection generation, validation, and
query-time runtime loading.

### Input Pinning

OpenAPI inputs must be reproducible. Two users installing the same manifest must
import the same provider description and generate the same default projections.

- A surface descriptor must provide a retrieval location and a content hash.
- The retrieval location does not need to be immutable; `sha256` is the
  reproducibility contract.
- If fetched bytes do not match `sha256`, source add fails.
- Bundled/core sources should prefer provider-owned immutable URLs when they
  exist.
- When a provider does not offer an immutable historical OpenAPI URL, Coral
  should mirror the exact OpenAPI artifact at a Coral-owned artifact URL and pin
  that URL with `sha256`.
- User-installed/community sources may use mutable provider URLs, but they must
  still provide `sha256`.

There is no unpinned URL mode.

### Surface IDs and Operation IDs

Surface IDs are manifest-authored and source-local. They are used by projection
overrides and materialized artifacts. They must be stable across source
versions unless the surface is intentionally replaced.

Operation IDs are imported and surface-scoped:

- If an OpenAPI operation has `operationId`, use it after minimal YAML-safe
  normalization.
- If `operationId` is absent, generate a deterministic ID from method and path.
- Duplicate operation IDs within one surface are a hard import error.

Projection overrides match imported operations by `(surface, operation)`.
Authors do not need to list every operation. Overrides are sparse.

## Semantic IR Requirements

The semantic IR is the normalized contract between importers and projection
machinery. It exists so projection generation can treat heterogeneous surfaces
with one vocabulary. It does not erase protocol-specific execution details.

The IR must represent:

- surfaces and their type,
- operations scoped to one surface,
- operation inputs and whether they are required,
- normalized scalar/object/list/map/enum/type shapes,
- output cardinality: none, singleton, list, wrapped list, or unknown,
- entity candidates scoped to their importing surface,
- relationship hints where they are discoverable,
- identity key evidence where it is discoverable,
- pagination capability and result-window limits,
- deprecation and description metadata,
- protocol-specific execution attachments needed by physical planners.

The first-wave OpenAPI importer must include REST execution attachments for:

- HTTP method,
- path template,
- path/query/header/cookie parameter serialization,
- request body shape,
- response status/media-type behavior,
- row extraction for list and wrapped-list responses,
- pagination behavior sufficient for the GitHub issues slice,
- surface-local error metadata where needed.

The IR should be serializable to a human-inspectable local artifact, preferably
YAML, but the artifact is internal. It may evolve with importer version changes.

### Types

Support:

- scalar types: string, integer, float/number, boolean, ID, timestamp/date-time,
  URI/string-like provider scalars, JSON/any, null,
- enums and constrained values,
- object types,
- input object types,
- arrays/lists,
- maps/dictionaries,
- nullable vs optional semantics,
- descriptions and deprecation metadata.

The IR should be shaped so interfaces, unions, field arguments, and GraphQL
connection-like output can be added without forcing a wholesale replacement of
the projection generator.

### Entities and Identity

Entities in first-wave IR are imported, surface-scoped entity candidates. They
are useful for projection generation, column naming, detail hints, and future
relationship handling.

The importer must not automatically assert that an entity imported from one
surface is equivalent to an entity imported from another surface. For example,
a future GitHub REST `issue` entity and GitHub GraphQL `Issue` entity may
represent the same provider concept, but that equivalence requires explicit
curation in a later wave.

Identity keys are evidence. They may be represented in IR, but first-wave
runtime planning does not perform cross-surface identity lookups.

## Projection Generation and Overrides

Projection generation turns semantic IR into the SQL catalog Coral exposes.

Default policy for this wave:

```yaml
projection_policy:
  default: derive_read_operations
```

If `projection_policy` is absent, the default is `derive_read_operations`.

The default generator must:

- consider only read-only operations,
- expose supported OpenAPI `GET` operations,
- skip mutating methods and request-body operations unless a later explicit
  policy supports them,
- derive a stable SQL name from operation/entity metadata,
- derive table columns from the operation output shape,
- derive required filters or function arguments from required operation inputs,
- derive pagination settings from imported surface metadata,
- generate a table for list-like and singleton readable resources when inputs
  can be represented as filters,
- generate a table function for search-like or operation-like reads when
  function arguments are the better SQL affordance,
- hide unsupported or unsafe operations by default with diagnostics.

Projection overrides are sparse patches over generated projections. They can:

- rename a projection,
- hide or publish a projection,
- choose `table`, `table_function`, or `search` when the generator's default is
  not the desired SQL affordance,
- set descriptions and guides,
- include, exclude, rename, or mark columns hidden,
- refine column type/nullability when the provider description is too vague,
- expose an operation input as a required filter, optional filter, or function
  argument,
- add search limits and detail hints,
- group or tag generated projections for docs/catalog purposes.

Projection overrides must not:

- define provider request paths, methods, parameter locations, response paths,
  or pagination mechanisms,
- introduce projections for operations missing from materialized IR,
- silently change the physical surface type,
- expose mutating operations in the first wave.

Materialized projection catalog entries should be normalized into a shape close
to the current runtime registration model:

- SQL table or table-function name,
- description and guide,
- surface ID,
- operation ID,
- generated/overridden provenance,
- input exposure model,
- output columns,
- pagination/fetch-limit behavior,
- search limits and detail hints,
- visibility status,
- link to protocol-specific execution attachment.

The runtime should consume the materialized projection catalog rather than
regenerating projections at query time.

## Materialization and App State

Source add is the materialization boundary.

For a v4 source, `coral source add` or imported source add must:

1. Parse and validate the manifest.
2. Validate source input declarations and user bindings using the existing
   source lifecycle path.
3. For each surface, fetch/read the pinned upstream description.
4. Verify the content hash.
5. Import semantic IR.
6. Generate default projections.
7. Apply projection overrides.
8. Validate generated projections against the IR.
9. Write materialized artifacts under the existing source app-state directory.
10. Persist source config and secrets only if materialization succeeds.

Normal query-time source loading must:

- resolve the manifest using the existing bundled/imported source rules,
- read materialized v4 artifacts from app state,
- validate lookup keys and importer version,
- fail clearly if required artifacts are missing or stale,
- avoid network access,
- avoid re-fetching upstream descriptions,
- avoid re-running importers or projection generation.

Proposed app-state paths:

```text
workspaces/<workspace>/sources/<source>/manifest.yaml
workspaces/<workspace>/sources/<source>/secrets.env
workspaces/<workspace>/sources/<source>/materialized/v4/manifest-fingerprint.yaml
workspaces/<workspace>/sources/<source>/materialized/v4/surfaces/<surface-id>/semantic-ir.yaml
workspaces/<workspace>/sources/<source>/materialized/v4/surfaces/<surface-id>/source-document.yaml
workspaces/<workspace>/sources/<source>/materialized/v4/projections.yaml
```

Bundled sources still resolve their manifest from the bundled catalog, but their
materialized artifacts live under the same installed source directory after
source add.

Materialized artifacts are looked up by:

- workspace,
- source name,
- source version,
- manifest fingerprint,
- surface descriptors including URL and `sha256`,
- importer version,
- projection generator version.

Changing any of those inputs makes the old materialization invalid for normal
load. The error should point users to reinstall or refresh the source.

### Refresh

Add a user-facing refresh path:

```text
coral source refresh <source>
```

Refresh must:

- read the installed source manifest,
- re-fetch pinned surface descriptions,
- verify they still match manifest `sha256`,
- import into a temporary materialization directory,
- regenerate and validate projections,
- atomically replace materialized artifacts only after all surfaces pass.

If fetched bytes do not match the manifest hash, refresh fails and reports the
observed hash. It must not silently update the manifest pin.

## Runtime Requirements

DSL v4 adds a new runtime path beside existing v3 backend dispatch.

Target path:

```text
DSL v4 installed source
  -> resolve manifest + materialized IR + projection catalog
  -> validate materialization fingerprint
  -> compile projection catalog
  -> dispatch each projection to a surface-specific physical planner
  -> register DataFusion tables/functions
```

The first physical planner is OpenAPI/REST. It should reuse existing HTTP
runtime pieces where practical:

- source variables and secrets,
- header auth and custom auth shapes,
- request headers,
- rate-limit handling,
- JSON fetch and row conversion helpers,
- pagination helpers,
- DataFusion table/table-function registration patterns.

The v4 runtime must not require the existing `HttpSourceManifest` table/function
authoring shape. Reuse helpers, not the v3 manifest model as the v4 contract.

## Representative Acceptance Slice

### GitHub REST via OpenAPI

Driven by a pinned GitHub REST OpenAPI description, the v4 source must import
and represent:

- list repository issues,
- search issues,
- get issue,
- at least one mutating operation as non-published IR evidence,
- at least one no-body success operation as non-published IR evidence.

The executable first slice must publish and run:

- repository issues as a generated or overridden SQL table,
- search issues as a generated or overridden table function with `kind: search`,
- get issue as a generated or overridden detail table/function.

The slice must exercise:

- required path parameters exposed as filters or function arguments,
- optional query parameters,
- singleton response,
- list response,
- wrapped list response through `items`,
- page/per_page or Link header pagination sufficient for the GitHub endpoints,
- generated columns from OpenAPI schemas,
- sparse projection overrides for names and column curation.

### No Network on Query Load

After source add succeeds:

- running a query must not fetch the OpenAPI document,
- running a query must not invoke the importer,
- running a query must not regenerate projections,
- missing/stale materialized artifacts must fail with a clear source lifecycle
  error.

### GraphQL Fast-Follow Readiness

No GraphQL importer or executor ships in this wave. However, the semantic IR
and projection generator boundaries must not assume OpenAPI is the only possible
surface forever. The IR should leave space for:

- root fields with arguments,
- nested selection policy,
- Relay-style connections,
- unions/interfaces,
- partial-data/error policy,
- GraphQL-specific execution attachments.

## Per-Step Acceptance Criteria

### 1. Scope

- This PRD states that the IR is for projection generation and validation, not
  for hiding physical execution differences.
- It states that v4 does not use top-level `backend`.
- It states that projections are generated by default and curated through
  overrides.
- It is written from the current v3 system baseline.

### 2. V4 Spec Models

- Add typed v4 manifest models with source common fields, `surfaces`,
  `projection_policy`, and `projection_overrides`.
- Existing v3 models remain unchanged for current manifests.
- Source-level inputs still use the existing input declaration model.
- Surface-level auth/header/rate-limit shapes reuse existing compatible spec
  types where practical.

### 3. Schema and Parser

- Existing parser tests and source specs keep passing.
- `dsl_version: 3` still requires top-level `backend`.
- `dsl_version: 4` rejects top-level `backend`, `tables`, and `functions`.
- `dsl_version: 4` requires at least one surface.
- Schema errors identify malformed surfaces and projection overrides clearly.

### 4. Semantic IR

- Add serde-friendly IR structs/enums.
- Round-trip representative REST model fragments.
- Validate unique surface IDs, unique operation IDs within a surface, valid type
  references, valid operation input references, and valid projection references.
- Entity candidates remain surface-scoped.
- REST execution attachments are present but separated from normalized
  projection-facing fields.

### 5. OpenAPI Importer

- Imports GitHub-shaped OpenAPI fixtures without network access in unit tests.
- Handles path/query/header/cookie parameters needed by the acceptance slice.
- Handles singleton, list, and wrapped-list responses.
- Detects pagination for the acceptance slice.
- Assigns deterministic operation IDs.
- Emits diagnostics for skipped unsupported operations.
- Refuses unpinned remote inputs.

### 6. Projection Generation

- Generates projections without author-provided projection entries.
- Produces stable names, columns, inputs, and visibility across runs for the same
  manifest, source document, importer version, and generator version.
- Skips unsupported or mutating operations by default.
- Generates usable projections for GitHub issue list, search, and get.
- Produces diagnostics explaining skipped operations.

### 7. Projection Overrides

- Overrides can rename, hide/publish, change projection kind where supported,
  curate columns, curate input exposure, set descriptions/guides, and add
  search/detail metadata.
- Overrides cannot invent operations or protocol execution details.
- Invalid overrides fail source add with errors naming source, surface,
  operation, and override path.

### 8. Materialization

- Bundled/core sources do not embed OpenAPI documents or pre-built IR in the
  Coral binary.
- Source add fetches/reads pinned descriptions, verifies hashes, imports IR,
  generates projections, applies overrides, validates artifacts, and writes
  materialized state.
- Existing source add rollback behavior also removes partial materialized v4
  artifacts.
- Query-time source loading performs no OpenAPI network I/O and no importer or
  projection generator execution.
- Missing/stale materialization errors point to source refresh or reinstall.

### 9. Refresh

- `coral source refresh <source>` exists.
- Refresh re-fetches pinned descriptions and verifies manifest hashes.
- Refresh writes into a temporary directory and atomically replaces artifacts
  only after full validation.
- Hash mismatch reports the observed hash and leaves existing artifacts intact.
- CLI/MCP/API docs are updated for the new command surface.

### 10. Runtime

- V4 runtime registers DataFusion tables/functions from materialized projection
  catalog entries.
- The OpenAPI/REST physical planner executes generated GitHub issue projections.
- Runtime errors reference source schema, projection name, surface ID, operation
  ID, and missing input where applicable.
- Existing v3 runtime behavior remains unchanged.

### 11. End-to-End

- GitHub issue list/search/get queries pass through v4 after source add.
- Tests prove query-time execution does not fetch OpenAPI or run importers.
- Tests cover required filters/args, optional params, row conversion,
  pagination, singleton/list/wrapped-list responses, and projection overrides.

### 12. Docs

- Document v4 manifest shape.
- Document surface pinning and Coral-owned artifact mirror policy.
- Document automatic projection generation and override limits.
- Document materialized artifacts and refresh.
- Document that v4 is OpenAPI-first and GraphQL is a fast-follow.
- Update docs in the same change as CLI/API/MCP surface changes.
